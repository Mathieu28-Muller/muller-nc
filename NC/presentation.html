Formulaire de déclaration Public — sans connexion

ISO §8.7.1 · §10.2.1

Le formulaire est accessible sans authentification. Il constitue le point d'entrée de toute non-conformité dans le système. Chaque soumission crée une NC au statut Ouvert et envoie un email de notification au pôle qualité.

ISO 9001:2015 §10.2.1 — Obligation L'organisme doit réagir à la non-conformité, en maîtrise les conséquences et évaluer le besoin d'actions pour en éliminer la cause. La déclaration documentée est la première étape de cette réaction.

Section Caractérisation du problème

Description du problèmeObligatoireISO §10.2.2
Champ le plus critique du formulaire. Décrire avec précision : symptôme observé, conditions d'apparition (fréquence, conditions environnementales), impact sur l'utilisation du produit.

À renseigner :
  • Quel est exactement le symptôme ? (message d'erreur, mesure fausse, composant cassé…)
  • Quand apparaît-il ? (systématiquement, aléatoirement, sous quelle condition)
  • Depuis quand ? Quel impact opérationnel ?
Ex : "Le banc de freinage VL affiche une erreur E304 lors de chaque test au-delà de 120 km/h simulé. Le problème est apparu après la mise à jour v2.4.1 il y a 3 semaines. Le centre technique ne peut plus effectuer ses contrôles techniques sur les véhicules rapides."
ISO §10.2.2 — Conserver informations documentées sur la nature de la non-conformité.
Photos / Vidéos du problèmeFacultatifFortement recommandé
Pièces jointes multimédias illustrant le problème. Le système accepte JPG, PNG, MP4, MOV jusqu'à 200 Mo par fichier. Depuis mobile, le bouton "📸 Prendre une photo" active directement l'appareil photo.
💡 Une photo vaut mieux qu'un long texte. Photographiez l'écran d'erreur, le composant défaillant, le résultat de mesure aberrant.

Suivi d'une NC existante (sans connexion)

Le formulaire intègre un module de suivi public. En saisissant son numéro de NC (format 260405-00001), tout déclarant peut consulter :

Les informations sensibles (analyse cause racine, coûts, actions internes) ne sont jamais visibles en accès public. Seul le statut et l'historique public sont exposés.

Profil Chef Produit / Pilote Rôle nc_chef_produit

ISO §10.2.1c · §10.2.1d · LNE
ISO 9001:2015 §10.2.1c — Obligation du pilote «Mettre en œuvre les actions nécessaires, y compris les actions correctives» — Le chef produit est responsable de la réponse aux actions CAPA qui lui sont assignées. Sa réponse documentée constitue une preuve d'audit.

Tableau de bord personnalisé du pilote

Le pilote dispose d'un tableau de bord filtré sur ses NC assignées uniquement. Il voit :
  • Le nombre d'actions CAPA en attente de sa réponse
  • Les NC où il est assigné comme pilote responsable
  • Les alertes de retard sur les actions non répondues
⚠️ Le pilote ne peut PAS créer de NC, changer les statuts, modifier la qualification, accéder à la configuration ou gérer les utilisateurs. Son périmètre est limité à la réponse aux actions qui lui sont assignées.

Répondre à une action CAPA

L'administrateur crée des actions et les assigne au pilote. Le pilote reçoit un email, accède à la fiche NC, et doit renseigner sa réponse.

⚡ Action Immédiate Réaction rapide pour contenir le problème. À effectuer dans les heures/jours suivant la NC.
Ex : mise en quarantaine du lot, retrait du produit, contournement provisoire
🔧 Action Curative Correction du problème sur le produit/client affecté. Traitement direct de la NC déclarée.
Ex : remplacement de pièce, mise à jour logicielle, recalibration
🔄 Action Corrective Élimination de la cause racine pour éviter la récurrence. Action la plus importante ISO.
Ex : modification du process de test, mise à jour procédure, formation
🛡 Action Préventive Prévention d'une NC potentielle identifiée par l'analyse. ISO §10.2.1e.
Ex : révision des points de contrôle, ajout d'un test automatisé
Champs obligatoires de réponse CAPA — ISO §10.2.2 Chaque réponse doit contenir : la description détaillée de l'action effectuée, la date de réalisation, et optionnellement des pièces jointes justificatives. Ces éléments sont conservés horodatés dans le journal d'audit.

Analyse 5 Pourquoi ISO §10.2.1b

La méthode des 5 Pourquoi permet d'atteindre la cause racine d'une NC en remontant la chaîne causale. Elle est accessible dans la fiche NC et peut être renseignée par le pilote ou l'admin.


NiveauQuestionExemple (banc freinage)
P1Pourquoi le problème est-il apparu ?Le capteur de vitesse retourne une valeur erronée
P2Pourquoi le capteur retourne une valeur erronée ?Sa plage de mesure est dépassée à haute vitesse
P3Pourquoi sa plage est dépassée ?La mise à jour v2.4.1 a modifié l'échelle sans validation
P4Pourquoi sans validation ?Pas de test automatisé sur les cas extrêmes
P5Pourquoi pas de test extrême ?Les plans de test ne couvrent pas les vitesses > 100 km/h
→ Cause racine identifiée : Plans de test incomplets, absence de couverture haute vitesse
→ Action corrective : Mettre à jour le plan de tests pour couvrir 0-180 km/h
ISO §10.2.1b — Évaluer le besoin d'actions pour éliminer la cause de la NC.

Diagramme Ishikawa (5M) LNE

Le diagramme causes-effet (Ishikawa) structure l'analyse en 5 familles de causes. Au moins une cause doit être renseignée pour valider l'analyse (obligation LNE).


CatégorieDescriptionExemple NC équipement
🧑 HommeCompétences, formation, erreur humaineTechnicien non formé sur la nouvelle version
🔩 MachineÉquipement, outil, maintenanceCapteur hors plage, usure prématurée
📦 MatièreComposants, matériaux, qualité fournisseurPièce de rechange non conforme fournisseur
📋 MéthodeProcédure, processus, instructionProcédure de mise à jour sans test validation
🌡 MilieuEnvironnement, conditions d'utilisationTempérature excessive dans le local de contrôle
LNE — Exigence diagnostic L'analyse Ishikawa documentée constitue la preuve que l'organisme a recherché et identifié les causes systémiques, non seulement le symptôme. Obligatoire pour le maintien des accréditations LNE sur équipements de mesure.

NC Parents — Groupement de non-conformités Admin uniquement

ISO §10.2.1 · LNE — Analyse systémique
Pourquoi les NC Parents ? Lorsque plusieurs NC distinctes partagent la même cause racine, les traiter isolément conduit à des actions correctives redondantes et à une vision fragmentée. La NC Parent permet de regrouper les NC liées sous une analyse unifiée — une seule cause racine documentée, un seul plan CAPA, une clôture cascade. Exigence implicite ISO §10.2.1e : «mettre à jour les risques et opportunités» en vision systémique.

Cycle de vie d'un groupe NC Parent

1. Créer le groupeSélectionner ≥ 2 NC
→ NC-P-XXXXX créée
→
2. RattacherAjouter satellites
tardifs si nécessaire
→
3. Analyser5 Pourquoi + Ishikawa
sur la NC Parent
→
4. ClôturerFermeture cascade
de tous les satellites
Règles invariantes (ISO §10.2.2) :
  • Numérotation NC Parent : NC-P-XXXXX (préfixe distinct)
  • Les NC satellites restent visibles dans la liste avec badge 📋 NC-P-XXXXX
  • La clôture du Parent clôture automatiquement tous les satellites (clôture cascade)
  • Le coût est porté par la NC Parent uniquement — les satellites n'ont pas de coût propre (éviter double-comptage)
  • L'email n'est PAS envoyé au pilote lors de la création du groupe (pas de spam sur regroupement)
  • L'historique (journal d'audit) de chaque satellite intègre les événements Parent (badge 📋)
  • Un satellite peut être rattaché tardivement à un groupe existant (fonction rattachement tardif)

Obligations clôture NC Parent Bloquant

Les champs suivants sont obligatoires et bloquants pour clôturer une NC Parent (contraintes plus strictes que les NC standards) :
  • 5 Pourquoi : P1 renseigné + Cause racine explicite
  • Ishikawa : au moins une cause dans une des 5 catégories (5M)
  • Type de cause : sélectionné parmi la liste configurée
  • Coût NC : renseigné (obligatoire sur NC Parent)
  • Commentaire de clôture : synthèse des actions menées
