1Contexte et objectifs

Le module Non-Conformités (NC) est actuellement hébergé sur un PC Windows local exposé via un tunnel Cloudflare sous le domaine formation-sav.fr. Ce domaine et cet hébergement sont utilisés comme environnement de développement (tests, évolutions futures, formation).

Objectif Déployer le module NC en PRODUCTION sur un VPS OVH dédié, sous un nouveau domaine professionnel Muller Automotive, accessible partout dans le monde depuis n'importe quel navigateur, sans installation, sans VPN.
Développement (inchangé)
formation-sav.fr/NC/
  • PC Windows local
  • Tests, mises à jour, nouvelles versions
  • Aucun impact sur les utilisateurs production
Production (nouveau)
espace-client.muller-automotive.fr
  • VPS OVH Linux — IP fixe
  • Usage quotidien entreprise
  • Clients, fournisseurs, auditeurs
  • SLA 99,9 % OVH garanti
⚠ Domaine de production (PLACEHOLDER) https://espace-client.muller-automotive.fr/NC/
Remplacer par le vrai domaine choisi par la direction IT avant déploiement.

7.4 Déploiement sur VPS OVH — 13 étapes
Pré-requis VPS OVH commandé · OS Ubuntu 22.04 LTS · Accès SSH root · DNS du domaine pointant vers l'IP du VPS
1
Connexion SSH et mise à jour OS
ssh root@<IP_VPS>
apt update && apt upgrade -y
2
Installation des composants serveur
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y nodejs nginx certbot python3-certbot-nginx rsync
npm install -g pm2
node -v   # doit afficher v20.x.x
3
Créer la structure de répertoires
mkdir -p /opt/nc/NC/media
chmod 755 /opt/nc /opt/nc/NC /opt/nc/NC/media
4
Transférer les fichiers depuis la clé USB
# Option A — rsync (Linux/Mac)
rsync -avz --exclude='node_modules' /media/usb/NC_PRODUCTION/ root@<IP_VPS>:/opt/nc/

# Option B — WinSCP (Windows, graphique)
# SFTP → <IP_VPS> → Login root → Copier E:\NC_PRODUCTION\ → /opt/nc/

# Option C — SCP (Windows ligne de commande)
scp -r "E:\NC_PRODUCTION\*" root@<IP_VPS>:/opt/nc/
5
Installer les dépendances Node.js
cd /opt/nc
npm install --production   # crée node_modules/ — durée 30s à 3min
6
Appliquer le schéma PostgreSQL et migrer les données
# Appliquer le schéma sur la base OVH
psql -U <PG_USER> -h <PG_HOST> -d nc_muller -f /opt/nc/schema_postgresql.sql

# Migrer les données JSON → PG (une seule fois au déploiement)
node /opt/nc/migrate.js --dry-run   # vérification sans écriture
node /opt/nc/migrate.js             # migration réelle

# Vérifier la parité complète
node /opt/nc/plan_controle.js       # doit afficher TOUT VERT ✅
7
Configurer et démarrer PM2
cd /opt/nc
pm2 start ecosystem.config.js
pm2 save
pm2 startup   # copier-coller la commande affichée
pm2 status    # "muller-nc" doit afficher "online"
pm2 logs muller-nc --lines 30
8
Configurer Nginx
nano /etc/nginx/sites-available/muller-nc
server {
    listen 80;
    server_name espace-client.muller-automotive.fr
                www.espace-client.muller-automotive.fr;
    return 301 https://$host$request_uri;
}
server {
    listen 443 ssl;
    server_name espace-client.muller-automotive.fr
                www.espace-client.muller-automotive.fr;
    root /opt/nc;
    client_max_body_size 200m;

    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 120s;
    }
    location /NC/media/ {
        alias /opt/nc/NC/media/;
        expires 30d;
    }
    location /NC/ {
        root /opt/nc;
        try_files $uri $uri/ =404;
    }
    location = / { return 302 /NC/; }
}
ln -s /etc/nginx/sites-available/muller-nc /etc/nginx/sites-enabled/
rm -f /etc/nginx/sites-enabled/default
nginx -t && systemctl reload nginx
9
Certificat HTTPS Let's Encrypt
Important : Le DNS doit pointer vers l'IP du VPS AVANT de lancer Certbot. Attendre la propagation (15–60 min).
certbot --nginx -d espace-client.muller-automotive.fr \
  -d www.espace-client.muller-automotive.fr
certbot renew --dry-run   # tester le renouvellement auto
10
Pare-feu et sécurité
apt install -y ufw fail2ban
ufw allow 22/tcp && ufw allow 80/tcp && ufw allow 443/tcp
ufw --force enable
systemctl enable --now fail2ban
sed -i 's/PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
systemctl reload sshd
11
Droits fichiers
chmod 600 /opt/nc/.env /opt/nc/ecosystem.config.js \
          /opt/nc/nc-data.json /opt/nc/nc-config.json
12
Sauvegardes automatiques (crontab)
mkdir -p /opt/backups && crontab -e

# Ajouter dans crontab :
0 2 * * * pg_dump -U <PG_USER> -h <PG_HOST> nc_muller | \
  gzip > /opt/backups/pg-nc-$(date +\%Y\%m\%d).sql.gz 2>/dev/null
0 3 * * * tar -czf /opt/backups/media-$(date +\%Y\%m\%d).tar.gz \
  /opt/nc/NC/media/ 2>/dev/null
0 4 * * * find /opt/backups/ -name "*.gz" -mtime +30 -delete
0 5 * * 1 certbot renew --quiet

14FAQ technique — Questions fréquentes
Quelle base de données est utilisée ?

PostgreSQL 16 — source de vérité unique depuis le 25/05/2026.

Le module fonctionne en mode DATA_SOURCE=postgres : toutes les lectures et écritures se font exclusivement dans PostgreSQL. Les fichiers JSON (nc-data.json, nc-users.json) sont conservés comme archives historiques uniquement — ne pas les modifier manuellement.

Sur OVH : recommandation Managed Database OVH PostgreSQL pour zéro administration + sauvegardes automatiques.

# Appliquer le schéma sur OVH (depuis le VPS)
psql -U <PG_USER> -h <PG_HOST> -d nc_muller -f /opt/nc/schema_postgresql.sql

# Migrer les données JSON → PG (une seule fois au déploiement)
node /opt/nc/migrate.js --dry-run   # vérification
node /opt/nc/migrate.js             # migration réelle

# Vérifier la parité complète
node /opt/nc/plan_controle.js       # doit afficher TOUT VERT ✅
Le code doit-il être réécrit pour Linux ?
Non. Node.js, Express, PM2, Nginx, PostgreSQL sont identiques Windows/Linux. Seuls les chemins changent (C:\formation → /opt/nc). Adaptations : ecosystem.config.js (chemins Linux + credentials PG OVH + SMTP), nginx.conf (nouveau domaine + HTTPS), URLs internes dans les HTML.
Comment déployer une mise à jour future ?

Développer et tester sur l'environnement dev (PC Windows + formation-sav.fr), puis :

scp C:\formation\server.js root@<IP_VPS>:/opt/nc/
scp C:\formation\NC\console.html root@<IP_VPS>:/opt/nc/NC/
pm2 restart muller-nc --update-env   # si server.js modifié
# Les fichiers HTML/JS sont actifs immédiatement sans redémarrage

Si le schéma PostgreSQL est modifié : ajouter le nouveau champ dans pgSyncFiche() ET dans schema_postgresql.sql, puis appliquer l'ALTER TABLE manuellement sur la base OVH.

Les données NC sont-elles perdues pendant la migration ?
Non. L'ancien PC reste actif pendant toute la migration. Les NC saisies entre le transfert clé USB et la bascule DNS doivent être récupérées manuellement (export CSV). Planifier la bascule un vendredi soir ou week-end pour minimiser le delta.
Comment gérer les utilisateurs NC en production ?

Depuis la console NC (nc_admin connecté) → ⚙ Configuration → Utilisateurs. Aucun accès SSH requis.

En cas d'oubli du mot de passe admin :

node -e "const bcrypt=require('bcryptjs'); \
  bcrypt.hash('NouveauPass',10,(e,h)=>console.log(h));"
# Mettre à jour le hash directement dans PostgreSQL :
psql -U <PG_USER> -h <PG_HOST> nc_muller
UPDATE nc_users SET password_hash='<HASH>' WHERE email='admin@...';
\q
